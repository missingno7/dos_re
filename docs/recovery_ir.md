# Recovery IR v0 — the shared representation every stage consumes

**Status: v0 LANDED (2026-07-17) — all §6 success criteria met on the
Lemmings pilot: `tools/irgen.py` regenerates the document deterministically;
`liftemit --from-ir` and `liftlink --from-ir` consume it through the shared
`dos_re.lift.ir` re-elaborator; the IR-path corpus (unlinked AND linked) is
byte-identical to the snapshot path, 212/212 modules; the demo corpus
replays ORACLE-CLEAN on the resulting graph; unsupported constructs land in
the IR's fail-loud ledger.  Originally drafted as:**  Companion to [`dos_re_2.0.md`](dos_re_2.0.md)
§1b: the system is built around one shared recovery IR + analyses + selected
emitters; every stage artifact is a projection of the IR.  This document
defines v0 — the smallest IR that lets the VMless emitter consume IR instead
of ad-hoc scan objects, and that the CPUless analyses can grow on without a
redesign.**

## 1. Why an explicit IR (and why now)

Today the pipeline's working representation is the in-memory pair
(`decode.Inst`, `cfg.FunctionScan`) plus side-channels (proof boards, link
reports, recovery facts).  That was enough to reach the full VMless graph,
but the CPUless stage needs liveness/stack/dataflow analyses that must NOT be
built by parsing emitted Python (banned by §1b), and the DOS-layout stage
needs memory-access facts collected in one place.  v0 makes the existing
knowledge explicit, serializable, and the SINGLE input to emission:

```
original binary + snapshot (code-byte authority)
→ decode + CFG recovery + effect classification     (existing machinery)
→ RECOVERY IR (this document)                        one file per program
→ analyses (v1+: liveness, stack effects, dataflow)  IR → annotated IR
→ selected emitter (VMless today; CPUless, bridge, diagnostics later)
```

Regeneration contract: the IR file is a generated artifact — deletable,
reproduced byte-identically from the binary + snapshot + facts + toolchain.
It carries the standard AUTOGENERATED header.

## 2. v0 scope (what the user-visible pipeline needs TODAY)

One JSON document per program (per code-byte snapshot), `recovery_ir.json`:

```jsonc
{
  "ir_version": 0,
  "provenance": {
    "exe": "<name + sha1 of the original binary>",
    "snapshot": "<id/path of the code-byte-authority snapshot>",
    "toolchain": "<dos_re commit>",
    "facts": "<sha1 of the recovery-facts file(s) consumed>"
  },
  "functions": {
    "1010:16A9": {
      "entry": "1010:16A9",
      "liftable": true,
      "refusals": [],                    // structured refusal records if not
      "exits": ["ret"],                  // exit-kind list: ret|retf|iret|jmp_far|jmp_ind
      "blocks": [                        // basic blocks, address-sorted
        {
          "leader": "16A9",
          "instructions": [
            {
              "ip": "16A9",
              "bytes": "c41ebe1f",       // raw bytes = ground truth + SMC guard input
              "kind": "seq",             // seq|call|call_far|call_ind|int|jcc|jmp|...
              "mnemonic": "les",
              // control flow (transfer kinds only):
              "target": null,            // near target ip, if static
              "far_target": null,        // [seg, off], if static far
              "int_no": null,
              // v0 effect summaries (coarse; v1 analyses refine them):
              "reads_mem": true,         // any memory read
              "writes_mem": false,       // any memory write
              "ports": null,             // {"in": [..], "out": [..]} static port refs
              "platform_effect": null    // recognized effect tag (see §4)
            }
          ]
        }
      ],
      "calls_near": ["16D4"],
      "calls_far": [["2911", "0559"]],
      "ints": ["21"],
      "signature": "c41ebe1f...",        // entry-byte SMC guard (emitter input)
      "max_iterations": 20000            // runaway budget (fact-overridable)
    }
  },
  "facts_applied": {                     // the recovery facts that shaped this IR
    "keep_interpreted": ["1010:03E7"],
    "boundary_heads": ["1010:46D7", "1010:0171", "..."]
  },
  "unsupported": [                       // fail-loud ledger: every construct v0 cannot
    {"entry": "....", "reason": "...", "detail": "..."}   // represent, NEVER silently dropped
  ]
}
```

v0 deliberately contains **no register/flag liveness and no inferred
signatures** — those are v1 ANALYSIS ANNOTATIONS layered onto the same
document (`analyses: {liveness: ..., stack_effects: ...}`), not a schema
change.  v0's job is to be complete and faithful, not clever.

## 3. The v0 pipeline changes

1. **`tools/irgen.py`** (new): binary + snapshot + entries + facts →
   `recovery_ir.json`.  Internally reuses `scan_function`/`decode` — the IR
   is their serialization plus effect tags and provenance.
2. **`lift/emit.py` consumes IR**: `emit_function_ir(ir_function, ...)` — the
   same emitter, fed from the IR record instead of a live `FunctionScan`.
   The existing scan-based entry point remains as a thin adapter
   (scan → IR record → emit), so liftverify/liftlink keep working unchanged
   while the migration completes.
3. **Equivalence gate**: regenerating the current VMless corpus THROUGH the
   IR must be byte-identical to the scan-path corpus (same emitted files),
   and the demo corpus must replay ORACLE-CLEAN — only then does the IR
   become
   the canonical path (dos_re_2.0.md §6 success criteria).
4. liftlink's edge computation moves onto the IR (`calls_near`/`calls_far` +
   exit kinds are already there), eliminating its private rescans.

## 4. Platform-effect tags (v0: recognize, don't yet bind)

v0 tags instructions whose effect class is already known so the adapter
binding stage (dos_re_2.0.md §4) has its work list in the IR:

| tag | recognizer (v0) |
|---|---|
| `int21_dos` | `int 21h` sites (filesystem/memory/console services) |
| `int10_video`, `int33_mouse`, `int16_kbd`, `int1a_time` | vector class |
| `int60_61_driver` | game-installed driver vectors (fact-assisted) |
| `port_pit` | in/out on 40h-43h |
| `port_vga_status` | in on 3DAh/3BAh |
| `port_vga`, `port_opl`, `port_sb`, `port_kbd`, `port_pic` | port ranges |
| `env_wait` | fact-declared wait loops (`keep_interpreted` / boundary heads) |

Tags are advisory in v0 (they change no emission); they become binding sites
for `platform.*` adapters at the effect-binding stage, and the scheduler-seam
recognition that dissolves the keep-interpreted queue starts from the
`env_wait` tags.

## 5. Non-goals of v0

- No SSA, no expression trees, no type inference — v1+ analyses own those.
- No 32-bit (PM) unification yet — `decode32`/`cfg32` get the same treatment
  after the 16-bit path proves the shape.
- No cross-function dataflow — function-local records only; the call graph
  is derivable from `calls_near`/`calls_far`.

## 6. Success criteria (v0 done means)

- `tools/irgen.py` regenerates `recovery_ir.json` deterministically from
  canonical inputs;
- the VMless emitter consumes the IR, and the IR-path corpus is
  byte-identical to the scan-path corpus for the Lemmings pilot;
- the demo corpus replays ORACLE-CLEAN on the IR-generated graph;
- unsupported constructs appear in the IR's `unsupported` ledger, loudly;
- liftlink computes edges from the IR without rescanning.
