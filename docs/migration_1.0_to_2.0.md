# Migrating a DOS_RE 1.0 project to 2.0

For projects started from the retired `template_dos_port` starter (or any
port built on the 1.0 manual-hook workflow).  Canonical target architecture:
[`dos_re_2.0.md`](dos_re_2.0.md); new-project path: [`getting_started.md`](getting_started.md).

## The conceptual migration

DOS_RE 1.0 recovered a game **one verified routine at a time**: run the
interpreter, hook a routine by hand, prove it, repeat — with the interpreter
as the permanent fallback for everything not yet recovered.  DOS_RE 2.0
inverts this:

```
old manually hooked project
  -> formal recovery facts          (game truths become versioned, evidence-backed data)
  -> shared recovery IR             (irgen: census + CFG + effects, one source of truth)
  -> generated lifted corpus        (liftemit/liftlink --from-ir; disposable output)
  -> strict VMless wall             (interpreter poisoned; zero interp_one sites)
  -> CPUless recovery               (de-carrier passes on the IR)
  -> DOS-layout dissolution         (native data structures)
  -> native runtime                 (platform adapters; oracle bridge optional)
```

The unit of progress is no longer "another routine recovered by hand" — it is
"another capability the pipeline gained" or "another recovery fact recorded."

## Practical mappings

| 1.0 artifact / habit | 2.0 replacement |
|---|---|
| Handwritten Python hook for a game routine | A CPUless authored override body plus generated backend adapters. Keep it separate from the reproducible baseline; classify it as faithful, enhancement, or behavioral. Emitter/linker workarounds still belong in tooling or recovery facts. |
| Interpreter fallback inside the recovered corpus | HARD-WALL VIOLATION: the strict runner poisons interpretation (`cpu.interp_forbidden`); uncovered addresses fail loud and become census/closure work |
| Project-local patch of framework behavior | Generic `dos_re` capability, contributed upstream (the framework must never import from a port) |
| Original EXE loaded at runtime by the "native" runner | Recovery-time input only: build a data-only boot image (`dos_re.bootimage`), boot EXE-free (`dos_re.independence`), lint + audit + clean-room test the independence |
| Manually maintained lifted/recovered code | Disposable generated output (AUTOGENERATED banner); one implementation, progressively lifted — lower stages become generated adapters |
| Separate "VMless", "CPUless", and "native" product runners | Use one profile-driven `play.py` and one planner. Record VM/CPU/ABI/memory properties per binding; produce standalone artifacts through the closed-world exporter. Old runners are temporary aliases only. |
| Per-routine ORACLE_PASSING proof as a gate for linking | Exact-interval oracle/candidate verification over a `ReplayArtifact` is authoritative; `replay.bisect_divergence` localizes a failing stable transition |
| Synthetic test drives (frame-counted key spam) | The recorded demo is the test — record real input once, replay it in lockstep |

## Suggested migration order

1. Point the project at a current `dos_re` toolchain (`irgen`, `liftemit`,
   `liftlink`, `bootimage`, and `independence`).
2. Record a real `ReplayArtifact` on the interpreted oracle; make it replay
   deterministically.  This is your safety net for every later step.
3. Separate each hand hook's semantic body from its CPU/register/stack wrapper.
   Put genuine program facts into `recovery_facts.json`, fix generic
   analysis/emitter gaps in the toolchain, and register any intentional
   handwritten implementation as an authored override. Never edit generated
   baseline output.
4. Assemble the full lifted graph, converge under the demo differential, and
   close the VMless wall (poison on, boundary heads as facts, zero
   `interp_one` sites — checked by `liftemit --require-vmless-wall`).
5. Sever the EXE: build the data-only boot image, switch the strict runner to
   it, and turn on the guards (`tools/lint_independence.py`,
   `tools/audit_boot_image.py`, a clean-room test).
6. Only then continue upward (CPUless).

The Lemmings pilot (`lemmings_port`) executed exactly this sequence; its
`docs/lemmings/native_pilot.md` ledger is the worked example, including the
dead ends.
